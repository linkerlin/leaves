## 〇、总纲

### 费曼式一句话

> leaves 是 **Go 生态里的 GBRT 推理与解释运行时**，并具备 **纯 Go 训练闭环**（T1–T5）：读 XGBoost/LightGBM 模型或自训练 → `ForestIR`/`ModelIR` → Native golden + Born 可选加速 → SHAP / 量化 / WASM 部署。

### 产品定位

| 维度 | 定位 |
|------|------|
| **核心身份（v1.x）** | Go GBRT **推理与解释**运行时 |
| **延伸身份（v2.x）** | 纯 Go **训练+推理一体化**（T1–T5 ✅） |
| **不是** | XGBoost 逐行克隆、Born 式通用 DL 框架 |
| **差异化** | 纯 Go 全链路 · XGBoost 3.x 互操作 · Tree SHAP · 训练→推理零切换 |
| **Born 角色** | 张量计算底座（CPU SIMD + WebGPU）；GBDT 领域逻辑仍在 `tree/`/`train/` |

### 核心命题

**如何在不丢弃生态兼容性的前提下，以 `ModelIR` 为统一 IR、`NativeEngine` 为 golden，用 Born 张量栈替换 GoMLX，演进为支持 XGBoost 3.x、可解释性、Windows 原生 GPU 的 GBRT 运行时？**

训练能力为 **v2.x 平行演进轨道**（见第十二章），不阻塞 v1.x 推理主线，但目标是在 Go 生态内实现**不依赖 Python/XGBoost 运行时**的独立完备训练闭环。

### 设计原则

1. **渐进增强，非推倒重来**：每个 Phase 产出可用的中间版本
2. **接口稳定性**：根包 `Ensemble` 公开 API 保持兼容，内部实现可替换
3. **Native 为正确性基准**：任何后端以 Native 为 golden；Born 是加速器，非真值来源
4. **能力探测优于用户强选**：`BackendAuto` 按模型特征与 workload 自动选择引擎
5. **IR 扩展，引擎跟随**：新特性先扩展 `ModelIR`/`ForestIR`，再更新引擎能力矩阵
6. **参考 XGBoost，不做克隆**：学习架构思想，用 Go 惯用方式表达
7. **训练与推理包分离**：`train/` 可依赖 `tree/`，`tree/` 不依赖 `train/`（Born 经 `tree/born_*.go` 直接读 `ForestIR`）
8. **测试驱动演进**：每个 Phase 有 golden 测试与真实 testdata 回归
9. **训练推理 IR 统一**：训练产出 `ModelIR`，`Fit → Save → Load → Predict` 必须 round-trip 可验证
10. **训练插件化**：Objective / Booster / TreeUpdater 注册式扩展，对标 XGBoost Registry

### 1.6 性能基准

单线程下，leaves Native 与 XGBoost C API 性能接近（差距 < 20%）。多线程持平。瓶颈在树遍历的标量循环。

**Born 现实预期**（参考 born `decode` 包经验：小算子场景 SimpleGo 常优于 XLA）：

| 场景 | Native | Born CPU | Born WebGPU（B4） |
|------|--------|----------|-------------------|
| 在线推理 batch ≤ 8 | ★★★ 最优 | ★★☆ 慢于 Native | ★☆☆ 不推荐（固定开销） |
| 批处理 100–1K | ★★☆ | ★★★ | ★★★ 目标场景 |
| 大 batch 10K+ 稠密 | ★☆☆ | ★★☆ | ★★★ 目标场景 |
| CSR 稀疏 | ★★★ | ★☆☆ densify | ★☆☆ |
| WASM 浏览器（js） | ★★★ Native | N/A | N/A（Born 不可用） |

### 2.2 顶层 IR（ModelIR）

不再将所有模型强行塞入 `ForestIR`：

```go
// model/ir.go
package model

type ModelKind int
const (
    KindGBTree ModelKind = iota
    KindDART
    KindGBLinear
    KindSklearnGBDT
)

type ModelIR struct {
    Kind        ModelKind
    NumFeatures int
    Output      OutputSpec   // groups, transform, base_score（可向量）
    Booster     BoosterIR    // 接口
}

type BoosterIR interface {
    Kind() ModelKind
    NEstimators() int
}

// tree/ir.go — 现有 ForestIR 扩展
type ForestIR struct {
    NumFeatures     int
    NumOutputGroups int
    Trees           []TreeIR
    BaseScore       float64      // 标量；向量见 BaseScores
    BaseScores      []float64    // XGBoost 3.1+ 向量 base_score
    WeightDrop      []float64
    AverageOutput   bool
    TreeInfo        []int        // 每棵树所属 output group
    IterationIndptr []int        // boosting 层 → 树索引边界
    NumParallelTree int
    CatEncoder      *CatEncoder  // 模型内分类编码（XGBoost 3.1+）
    Name            string
}

// linear/ir.go — 新增
type LinearIR struct {
    Weights []float64  // 按 XGBoost gblinear 布局
    Bias    []float64
}
```

### 2.3 预测语义（PredictRequest）

一次性定义，避免 Phase 2/3 反复改 API：

```go
// predict/request.go
package predict

type OutputKind int
const (
    OutputValue OutputKind = iota  // 变换后输出
    OutputMargin                   // raw score（树累加 + base_score）
    OutputLeaf                     // 叶索引
    OutputContribution             // Tree SHAP
    OutputApproxContribution       // Saabas 近似
    OutputInteraction              // SHAP 交互值
)

type IterationRange struct {
    Begin int // boosting 层起始（含）
    End   int // 结束（不含）；0 表示全部
}

type Request struct {
    Matrix      mat.Matrix     // Dense | CSR
    Output      OutputKind
    IterRange   IterationRange
    NEstimators int            // 兼容旧 API；映射到 IterRange
}

type Predictor interface {
    Predict(req Request, out []float64) error
}
```

### 2.4 引擎选择与加载 API

后端在**引擎创建时**选定（Born 按 shape 编译），非每次 `PredictDense` 切换：

```go
// io/io.go
type LoadOptions struct {
    LoadTransformation bool
    Backend            Backend  // Native | Born | SimpleGo | Auto
}

// BackendAuto 决策逻辑
func SelectBackend(ir *model.ModelIR, hint WorkloadHint) Backend {
    if !bornEngine.Supports(ir) { return BackendNative }
    if hint.Target == WASM { return BackendSimpleGo }
    if hint.BatchSize >= 256 && ir.IsNumericOnly() && hint.HasGPU {
        return BackendBorn
    }
    return BackendNative
}

// 用法（v1.0 目标）
m, _ := io.LoadFromFile("model.ubj", &io.LoadOptions{Backend: io.BackendAuto})
_ = m.Predict(predict.Request{Matrix: dense, Output: predict.OutputMargin}, preds)
contrib, _ := m.Explain().TreeSHAP(features)
```

根包旧 API 保留：

```go
model, _ := leaves.XGEnsembleFromFile("model.json", true)
model.PredictDense(vals, rows, cols, preds, 0, 4)  // 不变
```

## 三、Born 技术基础评估

> **决策（2026-06-13）**：废弃 GoMLX、gogpu 直连；**全面迁移至 Born** 作为 leaves 唯一张量计算底座。

### 3.1 核心模型

Born 采用**即时求值张量 + 可插拔 Backend** 范式（类 PyTorch/Burn）：

```
tensor.FromSlice → Backend.Add/Gather/Where → Data()
         │
         ├── backend/cpu   — 纯 Go + SIMD
         └── backend/webgpu — gogpu/wgpu，Windows DX12，零 CGO
```

**关键事实**：Born 是 DL 框架，**无 GBDT 抽象**；leaves 在 `tree/born_*.go` 用 Born 张量原语实现 GBDT 遍历，**直接读 `ForestIR`**（无 `TreeData`/`ForestData` 快照层）。

### 3.2 与树推理相关的 Born 原语

| Born 操作 | 在树推理中的用途 |
|-----------|----------------|
| `Lower` / `GreaterEqual` | 特征值与阈值比较 |
| `Where` | 路由左/右子树、NaN 分支 |
| `Gather` | 查节点表、取叶子值 |
| `SumDim` + `Mul` + `Expand` | 特征取值（one-hot trick） |
| `Cat` | 多树输出拼接 |
| `Argmax` | hist 增益扫描最佳分裂点 |
| `FromSlice` / `Data` | 与 `[]float64` 互操作 |

### 3.3 后端矩阵

| Born 后端 | 加速器 | 部署场景 | 对 leaves 的价值 |
|-----------|--------|----------|-------------------|
| `backend/cpu` | SIMD | 默认、CI、全平台 | 大 batch CPU 加速、正确性对照 |
| `backend/webgpu` | DX12/Vulkan/Metal | Windows/Linux GPU | **T4 GPU hist + 大 batch 推理** |
| autodiff | — | DL 训练 | **不用**（GBDT 分裂是离散搜索） |

### 3.4 限制与对策

| 限制 | 影响 | 对策 |
|------|------|------|
| 无 `CumSum` 原语 | hist 增益扫描 | `tree/born/cumsum.go` 用 `SumDim` 或 CPU 前缀和 |
| 主 dtype `float32` | 与 XGBoost float64 精度差 | 推理用 `float64` 路径；parity 容差文档化 |
| WebGPU 仅 `//go:build windows` 公开包 | Linux GPU 需 internal | 平台 build tag；训练 WebGPU hist 同路径 |
| Born API 演进快 | 升级风险 | `replace` 锁本地版本；Born 逻辑在 `tree/born_*.go`，无额外 facade 包 |
| 无 GBDT | 领域逻辑在 leaves | 保持 `ForestIR`；Born 只做张量 |

### 3.5 变换函数：不进张量图

与 XGBoost 两步语义一致：`margin = Σ(tree_leaf) + base_score` → `ObjFunction::PredTransform`。

- Born 图输出 **raw margin**
- CPU 侧 `transform.ApplyTransform*`（与旧 Born 引擎相同）
- **不在张量图中做 Softmax/Sigmoid**

## 十二、训练能力专项演进（独立完备）

> **本章定位**：与第一至十一章的推理主线**平行演进**，在 v1.2 推理能力稳定后启动。  
> **目标**：在 Go 生态内实现**不依赖 Python、不依赖 libxgboost CGO** 的 GBRT 训练全链路，并与现有 `ModelIR` / 推理引擎无缝衔接。

### 12.0 「独立完备」的定义

满足以下全部条件，方视为训练能力「独立完备」：

| 条件 | 说明 |
|------|------|
| **零外部 ML 运行时** | 训练、保存、加载、预测均在 leaves 进程内完成；无需 `python -m xgboost` 或 `libxgboost.so` |
| **IR 统一** | 训练产出 `ModelIR`（`ForestIR` / `LinearIR`），与推理侧同一数据结构 |
| **Round-trip 可验证** | `Fit → Save → Load → Predict` 闭环测试通过；与训练中途 `Predict` 一致 |
| **格式自洽** | 自有 `leaves.json` 为主格式；可选导出 XGBoost 3.x JSON 供外部生态 |
| **可观测** | 训练过程有 metric 日志、回调、checkpoint；支持早停与 CV |
| **可扩展** | Objective / Booster / TreeUpdater 插件式注册，对标 XGBoost Registry |

**不追求**：与 XGBoost **bit-exact** 数值复现（分箱算法、浮点序差异允许偏差）；追求**算法等价 + 趋势一致**。

// booster/booster.go — 对标 XGBoost GradientBooster

### 12.6 树构建：Hist 算法要点

```
参考：C:\GitHub\xgboost\src\tree\updater_quantile_hist.cc
       C:\GitHub\xgboost\src\common\hist_util.h
```

训练一轮的树构建流程：

```
1. 对每列特征，用 QuantileSketch 计算 max_bin 个分位点
2. 初始化根节点统计量 (G, H, cover)
3. 广度优先/深度优先扩展：
   a. 对每个活跃节点、每个候选特征：
      - 构建直方图 hist[bin] = {sum_g, sum_h}
      - 扫描所有分裂点，计算 gain
   b. 选最大 gain 分裂；若 gain < min_split_gain 则变叶
   c. 叶值 = -G/(H+λ) * learning_rate
4. 输出 TreeIR（SplitFeature, Threshold, DefaultLeft, LeafValue, SplitGain, SumHess）
```

**关键超参**（`train.Config`）：

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `max_depth` | 6 | 最大深度 |
| `max_leaves` | 0 | 0=仅 depth 限制；>0 则 leaf-wise |
| `max_bin` | 256 | 直方图分箱数 |
| `min_child_weight` | 1 | 子节点最小 Hessian 和 |
| `gamma` | 0 | 最小分裂增益 |
| `reg_lambda` | 1 | L2 正则 |
| `reg_alpha` | 0 | L1 正则 |
| `subsample` | 1 | 行采样 |
| `colsample_bytree` | 1 | 列采样 |
| `learning_rate` | 0.3 | 收缩因子 |

    "learner_model_param": {
      "base_score": 0.5,
      "num_feature": 30,
      "num_class": 1
    },
    "gradient_booster": {
      "name": "gbtree",
      "model": {
        "gbtree_model_param": { "num_trees": 100 },
        "trees": [ "..." ],
        "tree_info": [0,0,...,1,1,...],
        "iteration_indptr": [0, 2, 4, ...]
      }
    },
    "objective": { "name": "reg:squarederror" },
    "feature_names": ["f0", "f1"],
    "feature_types": ["float", "c"]
  }
}
```

设计原则：

- 字段命名**向 XGBoost 3.x JSON 靠拢**，降低迁移成本
- `objective` 写入文件但**仅作文档**；推理只需 `booster` + `transform` 推断
- 训练产生的 `TreeIR.SplitGain` / `SumHess` 一并保存，供 SHAP / importance 直接使用

**Round-trip 测试**（CI 必跑）：

```
Fit → Model() → Save → LoadFromFile → Predict
                 └→ ExportXGBoostJSON → (外部验证可选)
```

### 12.10 Born 在训练中的角色（T4+）

```

  go build ./...
  → treebuilder/hist.go + hist_parallel.go（CPU 多线程）

  → treebuilder/hist_accel_webgpu_*.go（WebGPU hist，Windows）

| 分裂增益扫描 | ✅ CPU 多线程 | ✅ Born CPU / WebGPU batch |
| 树结构决策 | ✅ 离散 argmax | CPU |
| margin 预测（训练环） | ✅ | ✅ Born GPU ≥256 行（auto） |

## 十三、风险与对策

| 风险 | 等级 | 状态 | 对策 |
|------|------|------|------|
| **XGBoost 3.1+ 模型无法加载** | 🔴 高 | **已缓解**（持续监控） | gblinear JSON、gamma/poisson 变换、`model.cats`/`XGBTreesRaw` round-trip；CI smoke + parity 矩阵扩展 |
| **Born 小 batch 比 Native 慢** | 🟠 中 | **接受**（架构 trade-off） | `BackendAuto` 默认 Native；bench 门禁 + README 披露；小 batch 不走 Born |
| **Born API 不稳定** (v0.x) | 🟠 中 | **已缓解**（持续监控） | `replace` 锁版本；逻辑隔离于 `tree/born_*.go` 直读 IR |
| **xgLinear 桥接错误** | — | **✅ 已闭合** | `linear/NativeLinearEngine` + `bridge_linear_test.go` |
| **训练结果与 XGBoost 不一致** | 🟠 中 | **接受**（不追求 bit-exact） | T1–T5 趋势一致验收；文档说明分箱/浮点差异 |
| **训练范围膨胀** | 🟠 中 | **已缓解** | T1–T5 + v3.1 已交付；`TODO.md`「明确不做」清单约束后续范围 |
| **Born 训练加速收益不确定** | 🟡 低 | **已量化** | WebGPU 可选；中小数据默认 CPU hist；MSLTR/大规模 bench 已披露交叉点 |
| **Go 版本升级阻力** | 🟡 低 | **持续监控** | CI Go 1.26.0；常规兼容性回归 |
| **GPU XLA 部署复杂度** | — | **✅ 已废弃** | GoMLX/PJRT 路线已弃；Born WebGPU（B4）替代 |
| **贡献者生态不足** | 🟠 中 | **接受** | 每 Phase 独立可交付版本；非代码可消除之风险 |
| **SHAP 与 XGBoost 数值差异** | 🟡 低 | **接受**（已文档化） | 可加性测试；与 XGBoost `pred_contribs` 逐元素可能不同 |

> **图例**：**已闭合** = 根因已修；**已废弃** = 风险源已移除；**已缓解/已量化** = 对策生效但仍需维护；**接受** = 固有 trade-off，靠选型与文档管控；**持续监控** = 低概率高影响，靠 CI/回归覆盖。
