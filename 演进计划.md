## 一、总判断

### 费曼式一句话

> leaves 已经不是“还差多少能力才能可用”的项目，而是“主干能力已经成立，接下来如何把入口、扩展点、性能治理和文档治理做扎实”的项目。

### 当前定位

`leaves` 的核心价值已经成型：

- **纯 Go GBRT 训练 + 推理 + 解释闭环**
- **与 XGBoost / LightGBM / scikit-learn 的互操作**
- **Native golden + Born 可选加速**
- **WASM / 热更新 / 量化 / profile 这些部署级能力已具备最小可用形态**

因此，未来一年的主线不再是无节制扩功能，而是：

1. **收口入口与文档**：把“推荐路径”说清楚，把历史章节瘦身。
2. **补齐扩展点**：让 objective / metric / booster / tree method 的扩展方式更一致。
3. **把性能治理制度化**：让 `BackendAuto` 和 benchmark 不再依赖经验判断。
4. **明确互操作边界**：把支持、实验性、明确不做的格式范围写实。
5. **为 v2.1 级稳定发布做准备**：建立更清晰的兼容承诺与发布门槛。

## 三、当前状态压缩摘要

### 3.1 架构判断

当前主干架构已经稳定，推荐路径是清晰的：

```text
训练：data.Matrix → train.Learner → booster/treebuilder/objective → ModelIR / leaves.json / XGB JSON
推理：io.LoadFromFile → ModelIR → NativeEngine / BornEngine → predict.Request
解释：ModelIR / ForestIR → explain.TreeExplainer / model.PredictContribs
部署：Reload / quantize / wasm / embed http demo
```

这意味着：

- `ModelIR` 已经承担统一中间表示职责。
- `NativeEngine` 已经承担正确性基准职责。
- Born 已经从“实验路径”转成“受能力矩阵约束的可选加速路径”。
- 训练和推理已经共享同一套 IR，而不是两套互相漂移的模型表示。

### 3.2 已完成能力压缩表

| 维度 | 当前状态 | 结论 | 依据 |
|------|----------|------|------|
| **推理核心** | LGB / XGB / SK / leaves.json 已能加载；Dense / CSR / margin / leaf / DART / RF 已覆盖 | **进入维护期** | [`README.md`](README.md)、[`io/`](io)、[`tree/`](tree) |
| **训练主干** | `gbtree` / `dart` / `gblinear`，`hist` / `exact` / `gpu_hist`，排序 / survival / tweedie / checkpoint / Eval / 外存均已落地 | **进入维护期** | [`train/learner.go`](train/learner.go)、[`train/`](train) |
| **解释能力** | Tree SHAP、interaction SHAP、Saabas、importance、dump 已具备 | **差异化已形成** | [`explain/`](explain)、[`model/`](model) |
| **部署能力** | int8 量化、`Reload`、WASM demo、HTTP embed demo 已具备 | **MVP 已完成** | [`quantize/`](quantize)、[`examples/wasm/`](examples/wasm)、[`examples/http/`](examples/http) |
| **数据入口** | `LoadDataAuto` / `FromFileAuto` 已完成内容嗅探 | **推荐继续沿用** | [`data/sniff.go`](data/sniff.go)、[`data/fromfile.go`](data/fromfile.go) |
| **自动语义** | `AutoTransform` 默认开启，推理推荐路径已经切到 `LoadOptions` | **方向正确** | [`io/transform_auto.go`](io/transform_auto.go) |
| **回归与 CI** | 多 OS 全量测试、WASM 大小门禁、Born 小 batch 负向门禁均在 | **工程基线可靠** | [`.github/workflows/ci.yml`](.github/workflows/ci.yml)、[`docs/testdata-matrix.md`](docs/testdata-matrix.md) |

### 3.3 明确剩余缺口

| 缺口 | 当前判断 | 处理原则 | 依据 |
|------|----------|----------|------|
| **ONNX 导入** | 仍是占位能力 | 不承诺短期补齐，只在明确 op 子集后再做 | [`io/onnx.go`](io/onnx.go) |
| **scikit-learn 支持** | 仍属实验性，协议范围窄 | 保守维护，优先把边界写清楚 | [`README.md`](README.md) |
| **Objective / Metric 扩展点** | 已有注册表，但仍有 `switch` 回退逻辑 | 这是下一年主线之一 | [`objective/registry.go`](objective/registry.go)、[`metrics/registry.go`](metrics/registry.go) |
| **BackendAuto 选型** | 当前启发式简单，主要看 batch / GPU / categorical | 这是下一年主线之一 | [`tree/backend_select.go`](tree/backend_select.go) |
| **根包兼容层** | 仍保留旧入口，实际已逐步委托到新路径 | 不急着物理迁移，先做文档与兼容策略 | [`README.md`](README.md)、[`ensemble_delegate.go`](ensemble_delegate.go) |

### 3.4 当前结论

代码层面的主要矛盾已经不是“功能缺失”，而是四件事：

1. **入口表述仍分散**：根包兼容入口、新路径入口、README、TODO、NOTES、演进计划之间存在历史叙述重叠。
2. **扩展方式不完全一致**：objective / metric 已部分注册化，但多分类、排序、booster、tree method 还没有统一到同一种扩展哲学。
3. **性能判断仍偏经验**：`BackendAuto` 与 benchmark 规则可用，但离“可解释、可复现、可回归”的工程标准还有距离。
4. **互操作边界写得不够直白**：哪些格式是稳定支持、哪些是实验性、哪些明确不做，应该更前置地告诉用户。

## 五、未来 12 个月目标

未来一年不是追求“功能数量最大化”，而是追求下面五个结果：

| 年度目标 | 结果定义 |
|----------|----------|
| **入口统一** | 用户只需要记住一条推理主路径、一条训练主路径，一条兼容迁移说明 |
| **扩展一致** | objective / metric / booster / tree method 的扩展方式尽量统一成注册式或工厂式 |
| **性能可解释** | `BackendAuto` 有明确决策表、benchmark 基线和回归门禁，而不是只靠经验 |
| **互操作边界清楚** | README 与 roadmap 直接说明稳定支持 / 实验性 / 明确不做的格式和场景 |
| **发布更像产品** | 文档、测试、CI、示例、兼容说明同步推进，形成 v2.1 级稳定发布口径 |

## 六、未来 12 个月路线图

### Phase A（0–2 个月）：文档与入口收口

**目标**：先把“项目现在是什么、推荐怎么用、哪些是兼容层”讲清楚。

**交付**：

- 精简路线图、README、README.zh、TODO、NOTES 之间的重复历史叙述
- 在 README 中把推荐入口收敛成：
  - 推理：`LoadFromFile` + `LoadOptions`
  - 训练：`NewLearner` / `LoadDataAuto` / `NewLearnerFromModelAndData`
- 明确根包兼容层的角色：**保留兼容，不再作为新能力首发入口**
- 将 [`TODO.md`](TODO.md) 从“阶段史正文的一部分”恢复成“可重新拆解的执行 backlog 容器”

**验收**：

- README / README.zh / roadmap 之间不再互相引用已经失效的章节结构
- 推荐入口不超过两条主路径
- 新用户从 README 到 demo 不需要先读完整历史文档

### Phase B（2–4 个月）：扩展点收口与插件化补齐

**目标**：让未来新增目标函数、指标、booster 或 tree method 时，不再扩散到多个 `switch`。

**交付**：

- 完成 objective 注册表的剩余收口，重点覆盖多分类与排序目标构造路径
- 完成 metric 解析路径的统一，减少内置分支与特殊命名散落
- 评估 booster / treebuilder 是否需要补充正式 registry；若不做，也要给出明确边界和设计说明
- 为扩展作者补一份简短开发文档：新增 objective / metric / booster 需要改哪些位置

**验收**：

- 新增一个自定义 objective 或 metric 不需要复制现有 `switch` 逻辑
- 相关回归测试覆盖内置与自定义扩展场景
- 扩展边界在文档中能说明白，而不是靠读源码猜

### Phase C（4–7 个月）：BackendAuto 2.0 与 benchmark 治理

**目标**：把“什么时候选 Native / BornCPU / BornGPU”从经验法则变成可回归的工程规则。

**交付**：

- 基于模型特征、batch、稀疏度、GPU 可用性扩展 `BackendAuto` 规则
- 把当前仅有的负向门禁（小 batch 不选 Born）扩成更完整的性能基线文档和测试矩阵
- 为 benchmark 输出统一记录格式，便于后续 release 比较
- 文档中补充更直白的部署建议：
  - 小 batch 在线推理优先 Native
  - 大 batch 数值树优先 Born
  - WASM 一律 Native

**验收**：

- `BackendAuto` 规则能从文档直接解释
- benchmark 基线可以在多个版本间比较
- 新增性能改动时，能判断是收益、退化还是仅工作负载变化

### Phase D（7–9 个月）：互操作边界硬化

**目标**：不是盲目支持更多格式，而是先明确哪些格式值得长期承诺。

**交付**：

- 给 LGB / XGB / scikit-learn / leaves.json / ONNX 建立支持等级表：
  - 稳定支持
  - 实验性
  - 调研占位
- 优先补强“已承诺路径”上的报错、文档、回归样本，而不是同时摊大饼
- 对 ONNX 给出明确策略：
  - 继续只做占位并推荐转换
  - 或只支持极小 TreeEnsemble 子集
- 对 scikit-learn 保守处理：如果不扩协议范围，就把限制写前置

**验收**：

- README 与 roadmap 对支持边界表述一致
- 用户在加载失败时能得到可操作错误，而不是“读源码才能知道为什么不支持”
- 支持矩阵与 regression matrix 可以互相映射

### Phase E（9–12 个月）：v2.1 稳定发布准备

**目标**：把项目从“功能已成型”进一步推到“版本治理成熟”。

**交付**：

- 建立 v2.1 发布检查表：测试、文档、示例、兼容说明、benchmark 摘要
- 整理“推荐路径 vs 兼容路径”的迁移文档
- 明确哪些变化允许在 v2.x 内发生，哪些必须等到下一次破坏性版本
- 为 explain / quantize / wasm / train-from-model 等差异化能力补齐稳定用法示例

**验收**：

- 发布说明可以直接从文档与 CI 结果生成
- 用户能区分“推荐 API”“兼容 API”“实验性 API”
- 版本升级不再依赖口头背景知识

## 七、非目标与延后项

以下事项在未来一年内**默认不进入主线**，除非出现明确用户需求、外部依赖成熟度变化或专项资源投入：

- **分布式训练 / Rabit / Spark / Dask / Ray**
- **官方 serving 框架**（`examples/http` 继续作为 embed demo，不把本仓库变成 serving 平台）
- **内置超参搜索**（继续建议外部工具）
- **根包 IO 物理迁移**（先维护兼容层，不做破坏性搬迁）
- **Multi-output tree 训练**（推理已具备基础能力，但训练默认不排入主线）
- **完整 ONNX Graph 导入**（除非缩成极小且明确的 TreeEnsemble 子集）
- **为追求“功能对齐”而引入新的大而杂模型族**

可选延后项：

- explain 侧进一步做大模型性能优化
- 独立 serving 仓库或独立 demo 工程
- 更细粒度的 benchmark artifact 发布

## 八、里程碑与验收口径

| 时间窗 | 里程碑 | 出口条件 |
|--------|--------|----------|
| **2026 Q3** | 文档/API 收口完成 | README / README.zh / roadmap / TODO / NOTES 口径统一；推荐入口清晰 |
| **2026 Q4 前半** | 扩展点收口完成第一轮 | objective / metric 扩展逻辑明显减少散落分支 |
| **2026 Q4 后半** | BackendAuto 2.0 与 benchmark 规则成型 | benchmark 文档与门禁能解释后端选择逻辑 |
| **2027 Q1** | 互操作支持等级表发布 | 支持、实验性、占位能力边界明确 |
| **2027 Q2** | v2.1 级稳定发布准备完成 | 文档、测试、CI、示例、兼容说明同步收口 |

### 全年统一验收原则

1. **Native 继续是正确性 golden**。
2. **任何自动选择逻辑都必须能被文档解释**。
3. **新增能力先收口入口和回归，再扩面**。
4. **优先减少文档和代码中的历史包袱，而不是新增宏大叙事**。
5. **只有进入 regression matrix 的能力，才算真正进入维护面**。

---

## 结论

过去一个阶段，`leaves` 已经把“纯 Go GBRT 训练 + 推理 + 解释 + 部署 MVP”这件事做成了。未来一年的正确动作，不是继续把路线图写得更长，而是把现有能力变得更稳、更清晰、更可扩展、更容易让外部用户正确使用。
